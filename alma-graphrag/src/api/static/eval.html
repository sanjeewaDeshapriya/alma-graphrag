Step 1 — The query set & gold standard

Evaluation replays a fixed set of natural-language traveller questions against the same city hotel pool for every system, so any difference in scores reflects the retrieval method only. Each query ships with a gold standard: the set of hotels that should be returned. Gold is currently rule-based — a hotel is relevant iff it satisfies every objective constraint of the query (price / rating / star / travel-time / amenity). Human relevance labels supersede it per-query when gold_human.json is present.

Step 2 — The three systems being compared

All three retrieve from the identical candidate pool and return a ranked top-K list of hotel ids. They differ only in what evidence they can see.

Filter baseline

Traditional filter-and-sort — the real-world OTA behaviour. Applies the structured constraints parsed from the query, then ranks survivors by rating (cheaper first on ties).

VectorRAG baseline

Vector retrieval — TF-IDF cosine similarity between the query and each hotel's text (name, description, amenities, attractions). Lexical-dense; swappable for dense embeddings.

Feasibility-first weighted multi-hop retriever. Scores each hotel on a composite of five sub-scores whose weights adapt to the query intent and traveller profile.

Step 3 — How each ranked list is scored

Binary relevance (a hotel is either in the gold set or not), evaluated at K = 10. Every metric is computed per query, then averaged across all queries.

Precision@10

P@10 = (relevant hotels in top-10) / 10

Of what we showed, how much was right. Rewards a clean top-10.

Recall@10

R@10 = (relevant in top-10) / (all relevant)

Of everything relevant, how much we surfaced. Rewards coverage.

nDCG@10

DCG = Σ reli / log2(i + 1) · nDCG = DCG / ideal-DCG

Rank-sensitive: a relevant hotel at rank 1 is worth more than at rank 10. The headline metric.

MRR

MRR = 1 / (rank of first relevant hotel)

How quickly the first good answer appears. 1.0 means rank 1.
